# Database schema & ERD

Database-per-service (see sql/init.sql for full DDL):

- `exchange_users` — users, stocks (reference data)
- `exchange_orders` — orders, audit_log
- `exchange_portfolio` — wallet_accounts, holdings, transactions

## Schema decisions to defend

- **DECIMAL(18,4), never FLOAT/DOUBLE** for money — binary floats can't
  represent 0.1; rounding errors are unacceptable in finance. In Java:
  `BigDecimal` everywhere on prices, `long` for share quantities.
- **UUID order ids, generated by order-service** — the id must exist *before*
  the Kafka publish, and must be globally unique across service instances
  (AUTO_INCREMENT can't do that without coordination).
- **`version` columns** → JPA `@Version` optimistic locking, because wallet
  and holding rows are written concurrently by settlement consumers.
- **No cross-database foreign keys** — `user_id` in orders is a logical
  reference. Referential integrity across services is maintained by the
  application/events, the price of independent deployability.
- **Composite indexes match query shapes**: `(user_id, created_at)` for "my
  orders newest first", `(symbol, status)` for open-order scans,
  `(user_id, executed_at)` for statements.
- **CHECK constraints** (`filled_quantity BETWEEN 0 AND quantity`) — the DB is
  the last line of defense for invariants.
